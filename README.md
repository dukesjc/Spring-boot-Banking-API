## 이 프로젝트를 선택한 이유?
핀테크 산업이 어떻게 작동하고 설계가 얼마나 복잡한지에 대해 궁금했습니다. 또한 백엔드 엔지니어링(고수준 및 저수준)에 대한 내 지식을 개선하고 싶었습니다.
<br/>
## 은행 관리 시스템이란?
은행 관리 시스템은 금융 기관이 제공하는 서비스에 대해 비용을 지불하는 데 사용되는 웹 기반 애플리케이션입니다.<br/>
은행에 따라 은행 관리 시스템의 구성 요소가 다를 수 있지만, 일반적으로 시스템은 기본 거래, 대출, 모기지 등을 관리하는 핵심 뱅킹 시스템으로 구성됩니다.

## 시스템 요구사항

### 기능 요구사항
이러한 요구사항은 세 가지 액세스 수준으로 나뉩니다: 관리자 모드, 창구 직원 모드, 고객 모드. 이들은 은행 관리 시스템이 제공해야 하는 서비스를 설명합니다.
* 고객:
  * 로그인과 비밀번호로 로그인합니다.
  * 하나 이상의 계좌를 생성합니다.
  * 개인 정보를 업데이트합니다.
  * 잔액을 확인합니다.
  * 거래 내역을 봅니다.
  * 송금합니다.
  * 출금합니다.
  * 현금 입금합니다.
  * 계좌가 활성화되면 직불 카드를 신청합니다.
  * 자격 요건을 충족하면 대출을 받습니다.
  * 필요하면 불만사항을 제기합니다.
  * 계좌 설정 시 환영 이메일을 받습니다.
  * 생일에 생일 인사 메일을 받습니다.
* 직원/스태프:
  * 로그인과 비밀번호로 로그인합니다.
  * 새로운 은행 고객을 등록합니다.
  * 고객 정보를 봅니다.
  * 고객 계좌를 관리합니다.
* 관리자:
  * 로그인과 비밀번호로 로그인합니다.
  * 직원, 관리자, 고객 세부 정보를 봅니다.
  * 은행 지점 세부 정보를 추가 또는 업데이트합니다.
  * 관리자 세부 정보를 추가 또는 업데이트합니다.
  * 직원을 강등 또는 삭제합니다.

### 비기능 요구사항
비기능 요구사항은 시스템에 부과되는 제약 또는 요구사항입니다. 소프트웨어의 품질 특성을 설명합니다.

#### 보안
은행 관리 시스템은 악의적인 공격의 대상이 되는 것으로 잘 알려져 있습니다. 따라서 보안은 시스템의 주요 요구사항입니다. 데이터는 정기적으로 백업되어야 하고 시스템 시설과 멀리 떨어진 안전한 지역에 보관되어야 하며, 이 데이터에 대한 모든 액세스는 신뢰할 수 있는 보안 네트워크에서 이루어져야 합니다.

#### 데이터 무결성
데이터 무결성은 조직의 데이터의 정확성, 완전성, 일관성, 유효성을 보장하는 개념 및 프로세스입니다. 데이터 무결성은 필수적이며, 무시하면 조직에 많은 비용이 들 수 있습니다.

#### 성능
은행 관리 시스템은 다중 클라이언트 시스템으로 동시 호출 중에 초당 목표 거래 수를 실행하고 각 클라이언트에 대한 응답 시간 목표를 충족해야 합니다.

#### 가용성
시스템 접근성은 은행 영업 시간 동안 필요합니다. 시스템은 모바일 뱅킹 앱과 ATM을 제공하기 위해 연간 평균 99.999% 가용성을 유지하면서 24/7 접근 가능해야 합니다.

#### 사용성
고객, 창구 직원, 관리자는 모두 시스템에서 제공하는 고유한 그래픽 사용자 인터페이스를 가져야 합니다. 모든 시스템 인터페이스는 사용자 친화적이고 이해하기 쉬워야 합니다.

## 고수준 설계
### 아키텍처
이 프로젝트는 마이크로서비스 아키텍처를 위한 API 게이트웨이를 제공하며 반응형 Netty 및 Project Reactor를 기반으로 구축되었습니다.

### 마이크로서비스를 선택한 이유?
마이크로서비스 아키텍처는 작은 서비스 집합으로 서버 애플리케이션을 구축하는 접근 방식입니다. 각 서비스는 자체 프로세스에서 실행되며 프로토콜을 사용하여 다른 프로세스와 통신합니다. 마이크로서비스는 독립적으로 배포할 수 있는 여러 기반으로 구축된 애플리케이션을 구축하는 데 도움이 됩니다. 한 서비스의 장애가 다른 서비스에 영향을 미치지 않습니다.

## 저수준 설계

### API 게이트웨이
API 게이트웨이는 서버(역 프록시 역할)입니다. 이는 시스템으로의 단일 진입점입니다. API 게이트웨이는 클라이언트가 수행하는 모든 요청을 처리하고 요청을 적절한 마이크로서비스로 지정합니다.

#### 기능
* 요청 처리
* 인증
* 캐싱(Redis 및 동시 HashMap을 사용하여 구현)
* 속도 제한
* XSS 응답 필터
* 서킷 브레이킹(resilience4j 사용)
* 로드 밸런싱(개발 환경에서 Eureka 사용, 프로덕션 환경에서 Kubernetes 사용)

### 고객 서비스
이는 고객 프로필 및 계좌 생성을 담당합니다. 고객 프로필 생성 후 SendGrid API를 사용하여 고객에게 비동기적으로 메일을 통해 알립니다.

#### 기능
* 고객이 하나 이상의 계좌를 가질 수 있는 고객 계좌(들) 및 프로필 생성
* 고객 세부 정보 업데이트
* 직불 카드 요청
* 고객 GET 요청
* 고객 계좌 DELETE
* 계좌 생성 시 환영 메일 메시지
* 매일 한 번 실행되어 고객에게 생일 축하 메일을 확인하고 전송하는 작업

### 거래 서비스
이는 모든 거래(입금, 송금, 출금)가 발생하는 곳입니다. 입금 거래는 비동기적으로 처리되지만 다른 거래는 Netty 논블로킹으로 처리됩니다.

#### 기능
* 입금, 출금, 송금과 같은 거래 처리
* 해당 서비스가 다운되면 모든 거래에서 재시도
* 고객의 모든 성공한 입금에 대한 GET 요청
* 고객의 모든 실패한 입금에 대한 GET 요청
* 계좌 명세서 생성에 사용되는 고객의 모든 거래

### 대출 서비스
이 서비스는 대출 신청 및 처리를 다룹니다. 두 가지 형태의 대출이 있습니다: 은행 대출과 P2P 대출입니다.

#### 기능
* 대출 신청
* 대출 처리, 분석 및 승인
* 대출 상환
* 매일 한 번 실행되어 기한이 지난 대출을 자동으로 차감하는 크론 작업

### 문제 서비스
이는 고객 문제/불만사항을 관리합니다. 제기되면 권한이 있는 직원이 해결하고 수정할 때까지 **PENDING** 상태로 유지됩니다.

#### 기능
* 불만사항 제기
* 불만사항 검토
* 불만사항 수정
* 모든 미결 불만사항 확인
* 백프레셔

### 직원 서비스
이 서비스는 직원 및 관리자를 관리합니다. 새로운 지점, 부서 등의 생성을 관리합니다.

#### 기능
* 새 직원 등록
* 새 지점 생성
* 직원 GET
* 직원 DELETE
* 직원을 **관리자**와 같은 역할로 승격
* 높은 역할에서 낮은 역할로 강등

### 문서화
오픈 API 사양은 REST API에 대한 표준 API 설명 형식입니다. Swagger는 SmartBear가 제공하는 오픈 소스 도구로 이 사양을 구현합니다.

이 프로젝트에서는 Swagger 3(OPEN API)을 구현하여 문서를 생성합니다.

## 이 프로젝트에서 배운 도전과제
이 프로젝트의 주요 목표는 백엔드 엔지니어링에 대한 내 지식을 개선하고 미세 조정하는 것입니다.

도전 과제:
* 도메인 기반 접근 방식을 사용한 마이크로서비스 설계
* 이벤트 기반 아키텍처(마이크로서비스 간 통신을 위한 이벤트의 적절한 사용)
* 핀테크 산업에 대한 경험 부족으로 인해 대출 서비스에 대해 많은 연구가 필요했습니다.
* 속도 제한 및 캐싱 레이어 구현
* 마이크로서비스를 Istio와 통합

배운 것:
* 멱등성 있는 API 작성
* 디버깅의 기술
* Hibernate N+1 문제
* 데이터베이스 연결 풀링 및 성능에 미치는 영향
* Hibernate 사용자 정의 검증기 작성
* 크론 작업으로 작업
* Spring Boot를 사용한 이벤트 기반 설계 구현
* 시스템 설계 지식 향상
* 지연 시간 감소 및 처리량 증가
* JVM 메모리 관리 및 스레드 최적화
* 내결함성(서킷 브레이커, 재시도)
* 데이터베이스 엔지니어링(SQL 쿼리 최적화, 인덱스 및 전체 테이블 스캔 방지)
* 보안 모범 사례
* 모니터링 및 로깅
* 서비스 메시에서 마이크로서비스 배포

이 프로젝트를 시작하게 되어 기쁩니다. 제 지평을 넓혀주었습니다.
